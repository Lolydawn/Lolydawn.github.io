jwt相关漏洞

jwt安全

  • json web token
  • 常用于授权场景，一旦用户登陆，后续的每个请求都将包含jwt

构成

  • 由三部分构成，分别是头部header、载荷payload、签证signature

头部 header

  • 头部承载两个信息 分别是声明类型
    • jwt
  • 另一个是声明加密算法
    • 通常直接使用 HMAC SHA256
  • 完整的头部
  • 把头部信息用base64编码后 即投入使用

载荷 payload

  • 包含三个部分
    • 标准中注册的声明
    • 公共的声明
    • 私有的声明
  • 标准中注册的声明（建议但是不强制使用）
    • iss：jwt签发者
    • sub：jwt所面向的用户
    • aud：接收jwt的一方
    • exp：jwt的过期时间，这个过期时间要大于签发时间
    • nbf：定义在什么时间之前，该jwt都是不可用的
    • iat：jwt的签发时间
    • jti：jwt的唯一身份标识，主要用来作为一次性token标识，防止重放攻击
  • 公共的声明
    • 公共的声明可以添加任何的信息，一般添加用户的相关信息或其他业务需要的必要信息.但不建议添加敏感信息，因为该部分在客户端可解密。
  • 私有的声明
    • 私有声明是提供者和消费者所共同定义的声明，一般不建议存放敏感信息，因为base64是可解密的，意味着该部分信息可以归类为明文信息
  • 示例

进行base64加密 投入使用

签证信息

  • header(base64后的)
  • payload(base64后的)
  • secret
  • 这个部分需要base64加密后的header和base64加密后的payload使用.连接组成的字符串，然后通过header中声明的加密方式进行加盐secret组合加密，然后就构成了jwt的第三部分
  • 这三部分用.连接 即可形成完整的jwt字符串

爆破jwt密钥

  • 到jwt.io使用爆破出来的密钥secret1进行加密数据 把用户名进行替换

jwt标头参数注入

  • 根据规范，只有alg标头参数是强制的。但实际上，jwt标头通常包含其他几个参数。
    • jwk：提供表示密钥的嵌入式json对象
    • jku：提供一个url，服务器可以从中获取包含正确密钥的一组密钥
    • kid：提供一个id，服务器可以使用该id在有多个密钥可供选择的情况下识别正确的密钥

jwk参数注入自签名jwt

  • 直接使用jwt插件生成一个rsa密钥 无需选择密钥大小

  • 生成的新的jwt字符串 即可绕过jwt的校验

jku参数注入自签名jwt

  • bp靶场提供了一个server exploit 把格式复制上去 然后把刚刚复制的那个粘贴进body部分 点击store即可

  • 然后到repeter里替换kid和用户名

  • 这里没复现成功 大概率是bp盗版的问题 可以尝试使用社区版

kid参数注入自签名jwt

  • 服务器可能使用多个加密密钥来签署不同类型数据，而不仅仅是jwt。因此header中可能会含有一个kid（key id）参数，该参数可以帮助服务器识别在验证签名时使用的哪个密钥。
  • 验证密钥通常存储为jwk集。在这种情况下，服务器可以简单的查找与kid令牌相同的jwk。但是jws规范没有定义这个id的具体结构，可以是一个任意字符串，因此攻击者可以使用kid参数来指向数据库中的特定条目，甚至文件名。
  • 可以使用/dev/null进行验证，这是一个空文件 会返回一个空字符串
  • 新建对称密钥，并将生成的属性值替换k为base64编码的空字节AA==

  • 然后把kid改为/dev/null 把sub改为administrator 点击sign即可

执行算法混淆攻击

  • 步骤如下
    • 获取服务器公钥
    • 将公钥转换为合适的格式
    • 创建一个恶意jwt，其负载经过修改，alg标头设置为HS256
    • 使用公钥作为密钥，使用HS256对令牌进行签名
  • bp靶场中有服务器公钥泄露

  • 然后把生成的字符串复制 base64编码
  • 然后生成一个普通的 把kid换成base64编码后的那个字符串

  • 然后去发包 把alg改成HS256 把用户名替换 点击sign 发包即可
