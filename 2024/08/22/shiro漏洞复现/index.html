shiro漏洞复现

shiro550 （Apache Shiro <=1.2.4或者能获取到shiro-key）

环境准备

  • docker pull medicean/vulapps:s_shiro_1
  • docker run -d -p 8080:8080 medicean/vulapps:s_shiro_1

判断特征

  • 返回包中含有rememberMe=deleteMe字段

漏洞原理

  • shiro<1.2.4版本中，key硬编码在源码中，而shiro提供了一个rememberme功能，如果用户勾选rememberme，并且登陆成功，那么shiro会将用户的cookie值序列化，然后采用shiro-key进行aes加密，最后base64encode，存储到cookie的 rememberme字段中。
  • 当用户携带cookie访问服务端的时候，shiro就会对cookie中的rememberme字段进行base64decode–>aes decode–>反序列化
  • 由于aes是对称加密（key共用），因此如果攻击者获得了key，则可以根据以上的链去构造一个payload放到cookie的rememberme字段中进行恶意攻击

手工复现

  • java的攻击链构造之后再谈 这里先使用序列化代码生成工具 ysoserial.jar来复现

上面的重点是aes加密的过程

iv可以固定 因为aes加密的特性 iv是被加在密文前去传递的 对方收到密文后 也能提取出iv

iv = b’\x00\x01\x02\x03\x04\x05\x06\x07\x08\x09\x0a\x0b\x0c\x0d\x0e\x0f’

格式要求是16字节（128位）

mode的选择也是看shiro的配置文件里怎么设置的 默认是CBC 如果配置了别的 一般就是GCM（一般是比较新的shiro版本）

判断是否存在shiro漏洞/组件

  • 看着挺难记的 漏洞存在的判断主要还是依靠bp插件、利用工具来判断 环境复杂 用url-dns链来判断比较好

waf绕过

  • 添加垃圾字符（不被base64解码的字符，shiro在处理特殊字符的时候会替换为空）

  • tab空白字符

  • http请求类型置空、瞎写 （tomcat）可行

  • 可叠加使用

shiro-721

漏洞原理

  • 基于 aes-cbc的padding缺陷
  • 通过 合法cookie 去逆向 爆破 从而构造恶意cookie（需要登陆成功）

权限绕过

cve-2010

  • /./admin

shiro与spring的解析不同

  • 在shiro中 /a/b 和 /a/b/ 是两个不同路径
  • 而在spring中 认为这两个路径是相同的，即会被解析为相同的资源
  • 所以可以使用/a/..;/admin/去进行权限绕过
  • 或者/a;/../admin/
  • shiro会把;后面的全部去除 所以是/a 即有权限访问 但是springmvc会把/admin/留下 所以最终造成了权限绕过
  • 漏洞修复
    • 把处理函数requestrui()换成别的安全函数

CVE-2020

  • 访问/a/admin 无法访问
  • 但是/a/%3badmin 即可绕过权限认证
  • %3b即;的url编码

CVE-2022

  • 使用/a/a%0admin即可绕过认证
  • %0a为换行符
